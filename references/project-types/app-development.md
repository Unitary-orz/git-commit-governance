# App Development Projects

适用于常见的 Web、Desktop、Mobile、全栈 App 项目。

这类项目在 Vibe Coding 场景下，最容易失焦的地方，是把 commit 写成“代码怎么改了”，而不是“功能怎么变了”。更自然的写法，是先回到用户视角：这次变化最终落在哪个页面、流程、入口或产品能力上。

核心感觉可以保持一句话：

更适合非程序员的 Vibe Coding 场景，让 commit message 聚焦于「功能怎么变了」，不要停留在「代码怎么改了」。

## 推荐写法
- `summary` 语言：默认中文
- 提交格式：继续使用 `type(scope): summary`
- 选 `scope` 时，先看用户从哪里感知到变化，再看实现落在哪层代码

## 推荐 scope 来源

在通用 App 项目里，`scope` 往往更适合来自这些地方：

- 页面：例如列表页、详情页、编辑页、设置页、导入页、导出页
- 流程：例如登录、注册、导入、导出、结算、发布、同步
- 功能域：例如字段管理、模板管理、搜索、筛选、通知、权限、标签
- 任务域：例如邮件、翻译、开发、研究、审批、日程
- 仓库治理：例如发布、构建、脚本、CI、仓库配置

这些来源都比 `ui`、`api`、`frontend`、`backend`、`db` 更接近真实产品语义。用户在哪个地方感知到变化，`scope` 通常就可以靠近那个地方。

## 选择感觉
- 先看这次变化更像是在补一个入口、修一个流程、收敛一组规则，还是在维护仓库本身。
- 同一条用户流程跨了前后端、数据库、桌面壳或脚本也没关系；只要用户感知到的还是同一个流程，`scope` 仍然可以自然落在那个流程域。
- 页面名、流程名、功能域名，只要在项目里长期稳定、团队也看得懂，就很适合直接做 `scope`。
- 技术层名更适合留给代码和目录结构，不太适合放在 commit 历史里代表用户变化。
- 当一次提交主要是在收敛发布、构建、脚本、版本号、项目配置，回到 `release`、`build`、`ci`、`repo` 这类仓库域会更顺手。

## 更像产品语言的 scope

下面这些 scope 风格，通常比技术层命名更贴近 Vibe Coding 的阅读习惯：

- `list`
- `detail`
- `editor`
- `settings`
- `auth`
- `search`
- `filter`
- `import`
- `export`
- `template`
- `field`
- `tag`
- `billing`
- `release`
- `repo`

项目如果已经有自己稳定使用的一组 scope，优先沿用项目现成命名，不必为了通用参考重新改名。

## 不太有帮助的 scope
- `ui`
- `api`
- `frontend`
- `backend`
- `db`
- `data`
- `code`
- `system`
- `misc`

这些名字更像实现层切片。它们对写代码时有用，但放进 commit 历史后，往往很难快速读出“产品上到底变了什么”。

## 使用提示
- `summary` 更适合直接描述功能变化、交互变化、规则变化，而不是描述“抽了 hook”“改了 schema”“重构了组件”。
- 当一次改动横跨多个技术层，但最终只服务于一个用户入口时，让 `scope` 继续贴着那个入口，会让历史更稳定。
- 当一次改动明显同时包含两个独立功能点时，拆成两条 commit，往往比硬找一个大 scope 更清楚。
- 文档提交也可以沿用功能域 `scope`。例如更新导入说明时，`scope` 仍然可以是 `import`，而不是直接写成 `docs`。

## 示例
- `feat(import): 新增导入预览里的异常行筛选入口`
- `fix(search): 修复搜索后切换标签仍保留旧条件的问题`
- `refactor(field): 收敛字段编辑页的校验与保存规则`
- `docs(template): 补充模板继承规则与覆盖顺序说明`
- `feat(settings): 新增桌面端自动启动与静默启动开关`
- `fix(auth): 修复登录过期后返回原页面失败的问题`
- `feat(mail): 新增未回复邮件优先级判断与延迟回复规则`
- `refactor(dev): 收敛代码补丁生成前的检查流程`
- `chore(release): 同步桌面版版本号与发布说明`
- `chore(repo): 清理误导性的脚本描述与仓库配置`

## 反例
- `fix(ui): 调整样式和逻辑`
- `refactor(backend): 优化接口`
- `feat(api): 增加能力`
- `chore(system): 调整一堆配置`

更容易读懂的写法，通常会直接说清产品变化：
- `fix(export): 修复导出概览页的字段摘要错位`
- `feat(list): 新增批量选择后的继承操作入口`
- `refactor(import): 收敛字段映射兜底与冲突提示规则`
- `docs(settings): 明确同步开关与本地缓存行为`
