# git commit 提交说明

本文档用于统一 `<项目名>` 仓库的提交信息写法，便于按功能域追踪变更。

## 1. 提交格式

统一使用：

`<type>(<scope>): <summary>`

示例（参考风格）：

- `fix(import): 修复导入时字段映射兜底顺序`
- `docs(repo): clarify release branch handoff process`
- `feat(field): 收敛端口字段与应用关联流程`
  - `- 移除端口遗留字段，统一字段语义`
  - `- 重构编辑与关联入口，复用共享规则`

## 2. type 建议

- `feat`：新增功能
- `fix`：问题修复
- `refactor`：重构（不改外部行为）
- `docs`：文档变更
- `test`：测试补充或调整
- `chore`：构建、脚本、仓库杂项
- `build`：构建链路、打包产物、版本生成
- `ci`：持续集成与自动化流程
- `revert`：回滚已有提交

## 3. scope 约束

优先使用项目内已经约定的功能域或业务域 scope。

推荐写法：

- 使用对项目成员直接可读的功能域，如 `import`、`export`、`field`、`template`
- 一次提交只保留 1 个 scope
- 优先表达“这次改动服务于哪个功能域”，而不是“改了哪一层技术实现”

不推荐写法：

- `ui`、`api`、`backend`、`db` 这类纯技术层 scope
- `import-template`、`host-and-app` 这类拼接多个域的 scope

> 如项目已定义 scope 白名单，请在这里替换为项目自己的说明与列表。

## 4. summary 写法

- `summary` 的默认语言应跟随项目的 `summary_language` 偏好；未单独指定时，可默认使用中文动词短句。
- 优先描述功能点变化、用户可感知结果或明确的业务影响。
- 可以保留必要的代码层动作词，如“重构”“收敛”“统一”，但必须说清楚是围绕哪个功能点。
- 不要只写“修复问题”“优化代码”“调整逻辑”这类空泛表述。
- 保持单一主题，避免在一个 summary 混入多个无关改动。
- 单行或多行应优先跟随项目的 `body_style` 偏好；当同一目标下仍需要补充关键变化维度时，可以追加简短 body。

推荐：

- `feat(list): 新增应用列表批量继承入口`
- `fix(import): 修复导入时字段映射优先级与兜底匹配`
- `refactor(template): 统一模板应用流程的跳过与覆盖规则`
- `docs(field): 补充字段合并交互说明`

不推荐：

- `update code`
- `fix bug`
- `调整一下提交`
- `修复问题`
- `优化代码`
- `调整逻辑`
- `feat(list): 修复导入并更新模板并改字段`

## 5. 提交前检查

- 先确认这次提交表达的是一件清晰的事，而不是把多个无关改动打包到一起。
- 确认工作区和暂存区没有混入本次提交范围之外的文件。
- 默认检查暂存改动的整体范围；当改动较大、跨模块或仍有不确定性时，再看完整 diff。
- 如果发现无关改动，优先考虑拆分成独立 commit；若暂时保留在同一提交中，也应明确说明原因与边界。
- 如果当前任务只是帮助完成提交，默认不要改代码；若确实需要改，先确认。

常用检查命令：

- `git status --short`
- `git diff --cached --stat`
- `git diff --cached`

## 6. 特殊约束

- 不要使用 `--no-gpg-sign`
- 如果签名失败，先处理签名问题，不要绕过
- commit-only 任务默认不改代码；若确实需要改，至少先询问用户
- 如项目有发布、版本、桌面端、迁移脚本等特殊规则，请在这里补充
